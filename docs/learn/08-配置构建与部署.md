# 08-配置构建与部署

## 1. 这篇文档回答什么问题

- 配置从哪里来，如何覆盖？
- 项目如何构建与部署？
- CI/CD 现状如何？

## 2. 核心结论

- [事实] 配置读取顺序是“YAML + `GOIM_*` 环境变量覆盖”。
- [事实] 构建支持本地 `go build`（Makefile）和 Docker 多阶段构建（`deploy/Dockerfile`）。
- [事实] 部署样例为 Docker Compose（`deploy/docker-compose.yaml`）。
- [事实] 当前仓库未发现 CI/CD pipeline 配置文件（`.github/workflows`、`.gitlab-ci.yml` 等）。

## 3. 关键证据

- 配置加载：`internal/config/config.go`
- 本地配置：`config/go-im.yaml`
- 容器配置：`deploy/docker-config.yaml`
- 构建命令：`Makefile`
- 镜像构建：`deploy/Dockerfile`
- 部署编排：`deploy/docker-compose.yaml`

## 4. 详细说明

### 4.1 配置体系

| 层级 | 文件/来源 | 说明 |
|---|---|---|
| 默认本地配置 | `config/go-im.yaml` | 开发环境默认参数 |
| 容器配置 | `deploy/docker-config.yaml` | compose 挂载到容器内 `/app/config/go-im.yaml` |
| 运行时覆盖 | `GOIM_*` 环境变量 | 通过 Viper 自动覆盖（如 `GOIM_JWT_SECRET`） |

### 4.2 配置安全校验

- `cfg.Validate()` 在 `release` 环境下会强制：
  - `jwt.secret` 不能空或 `changeme`
  - `ws_server.internal_api_key` 不能为空
- 三个服务启动都会执行该校验（`cmd/*/main.go`）。

### 4.3 本地构建与运行

- 构建：`make build`（输出 `bin/im-api`, `bin/im-ws`, `bin/im-push`）
- 测试：`make test`（`go test ./... -v -cover`）
- 本地运行：`make run-api|run-ws|run-push`

### 4.4 Docker 构建与部署

- `deploy/Dockerfile` 使用多阶段构建：
  - 第一阶段统一编译三个二进制
  - 后续分别产出 `api/ws/push` 镜像目标
- `deploy/docker-compose.yaml` 负责编排：
  - mysql / redis / kafka
  - im-api / im-ws / im-push
  - 通过环境变量注入密钥与 DSN

### 4.5 CI/CD 与发布方式（不适用项说明）

- **当前不适用（仓库证据不足）**：
  - 未发现 `.github/workflows/*`
  - 未发现 `.gitlab-ci.yml`
  - 未发现 Jenkinsfile / Terraform / Helm
- 结论：仓库中仅能确认“本地/compose 部署路径”，无法确认正式发布流水线。
- 后续验证建议：
  1) 查组织级 CI 平台；
  2) 查镜像仓库发布记录；
  3) 向维护者确认发布 SOP。

## 6. 常见误区 / 风险 / 注意点

- 误区：以为改 YAML 就一定生效。若设置了 `GOIM_*`，会覆盖 YAML。
- 风险：`release` 模式密钥配置不完整会直接启动失败。
- 风险：WS 内部地址（`rpc_advertise_addr`）配置错误会导致 Push 无法在线推送。
