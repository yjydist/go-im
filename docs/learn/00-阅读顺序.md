# 00-阅读顺序

## 1. 这篇文档回答什么问题

- 第一次接手这个仓库，应该按什么顺序读？
- 不同目标（快速了解、改需求、排查问题）分别怎么读？

## 2. 核心结论

- [事实] 这是一个**三服务协作**的 Go IM 后端，先理解三入口，再看消息主链路，最后看 API 业务面。
- [事实] 建议主线：`cmd/*`（启动入口）→ `internal/ws`（消息入口）→ `internal/push`（异步投递）→ `handler/service/repository`（HTTP 业务）。
- [事实] 新人最容易卡住在：
  1) 只看 API，忽略 WS+Push 才是消息主链路；
  2) 把 ACK 当作“送达确认”；
  3) 不清楚 Redis 在线状态与 Kafka 消费重试关系。

## 3. 关键证据

- 三入口：`cmd/api/main.go`、`cmd/ws/main.go`、`cmd/push/main.go`
- 路由入口：`internal/handler/router.go`
- WebSocket 主逻辑：`internal/ws/client.go`、`internal/ws/server.go`
- Kafka 消费与推送：`internal/push/consumer.go`、`internal/push/pusher.go`
- 配置与环境：`internal/config/config.go`、`config/go-im.yaml`、`deploy/docker-config.yaml`

## 6. 常见误区 / 风险 / 注意点

- 误区 1：只看 `internal/handler`，会漏掉消息主链路（WS→Kafka→Push）。
- 误区 2：把 `ack` 当送达回执。当前 ACK 在 WS 入队后即返回，非终态送达确认（见 `internal/ws/client.go`）。
- 误区 3：忽略配置覆盖。`GOIM_*` 环境变量会覆盖 YAML（见 `internal/config/config.go`）。
