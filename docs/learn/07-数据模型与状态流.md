# 07-数据模型与状态流

## 1. 这篇文档回答什么问题

- 核心数据存在哪？结构是什么？
- 状态如何产生、变化、过期和清理？

## 2. 核心结论

- [事实] 持久化主数据在 MySQL（用户/好友/群组/消息/离线消息）。
- [事实] 运行时状态在 Redis（在线状态、消息去重、群成员缓存）+ 内存 Hub（当前进程在线连接映射）。
- [事实] 消息在 Kafka 中间流转，最终落地 MySQL 并按在线状态分发。

## 3. 关键证据

- MySQL 表结构：`deploy/mysql/init.sql`, `internal/model/*.go`
- Redis 键设计：`internal/repository/redis_repo.go`
- 在线状态写入/刷新/删除：`internal/ws/client.go`
- 离线消息写入/拉取/清理：`internal/push/pusher.go`, `internal/repository/message_repo.go`, `internal/service/message_service.go`

## 4. 详细说明

### 4.1 MySQL 关键表

| 表 | 用途 | 对应模型 |
|---|---|---|
| `users` | 用户账号信息 | `internal/model/user.go` |
| `friendships` | 好友关系（待确认/已接受） | `internal/model/friendship.go` |
| `groups` | 群组主表 | `internal/model/group.go` |
| `group_members` | 群成员关系 | `internal/model/group.go` |
| `messages` | 消息主表（含 `msg_id` 唯一） | `internal/model/message.go` |
| `offline_messages` | 离线索引表（user_id -> message_id） | `internal/model/message.go` |

### 4.2 Redis 键与语义

| 键模式 | 值 | 来源模块 | 作用 |
|---|---|---|---|
| `online:{user_id}` | `ws_rpc_addr` | `ws/client.go` | 记录用户在线节点地址 |
| `msg_dedup:{msg_id}` | `"1"` | `ws/client.go` | WS 消息幂等防重 |
| `group:members:{group_id}` | SET(user_id) | `ws/client.go` / `push/pusher.go` | 群成员缓存 |

### 4.3 消息状态流（简化）

1. 客户端上行消息进入 WS。
2. 通过 `msg_dedup` 检查，避免重复入队。
3. Kafka 异步传递到 Push。
4. Push 先写 `messages`。
5. 在线：调用 WS 内部接口推送。
6. 离线：写 `offline_messages`。
7. 用户调用 API 拉离线后，按 `maxOfflineID` 清理已拉取记录。

### 4.4 在线状态流

1. 连接建立：`SetOnline(user_id, ws_addr, ttl)`。
2. 心跳续期：`writePump` 定时刷新 TTL。
3. 连接断开：`DelOnline(user_id, expectedAddr)` 条件删除（防止旧连接误删新连接状态）。

### 4.5 当前可见的不一致点

- [事实] 群成员缓存 TTL 在 WS 回填中固定 30 分钟（`internal/ws/client.go`），在 Push 回填中来自配置 `push.group_member_cache_ttl_sec`（`internal/push/pusher.go`）。
- [推断] 这可能导致不同链路下缓存行为不一致，排障时要注意来源。

## 5. 如何继续深入

- 对照 [05-核心运行流程](./05-核心运行流程.md) 把每个状态点映射到函数。
- 如需改表结构，先读 [10-需求修改指南](./10-需求修改指南.md)。

## 6. 常见误区 / 风险 / 注意点

- 误区：只看 `messages` 表，不看 `offline_messages`，会误判“为什么用户没收到”。
- 风险：直接改 Redis 键名会导致 WS 与 Push 互相读不到状态。
- 注意：仓库运行代码中未发现生产自动迁移，表结构变更要同步 SQL 脚本与部署流程。
