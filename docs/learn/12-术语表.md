# 12-术语表

## 1. 这篇文档回答什么问题

- 文档和代码里出现的缩写、术语分别是什么意思？
- 这些术语在本项目里对应哪段实现？

## 2. 核心结论

- [事实] 本项目术语主要分为 4 类：
  1) 通用 IM/后端术语（如 ACK、TTL、幂等）；
  2) 本项目协议术语（`ChatData`、`PushMsg`）；
  3) 基础设施术语（Kafka offset、Redis key）；
  4) 代码分层术语（handler/service/repository）。
- [事实] 理解术语能显著降低误判，尤其是 ACK、离线消息、去重 key 三者关系。

## 3. 关键证据

- 协议定义：`internal/ws/protocol.go`
- 消息处理：`internal/ws/client.go`、`internal/push/pusher.go`
- Redis 键定义：`internal/repository/redis_repo.go`
- 配置结构：`internal/config/config.go`
- 错误码：`internal/pkg/errcode/errcode.go`

## 4. 详细说明

### 4.1 通用术语

| 术语 | 含义 | 本项目对应实现/证据 |
|---|---|---|
| IM | Instant Messaging，即时通讯 | `README.md` 项目描述 |
| REST API | 基于 HTTP 的接口风格 | `internal/handler/router.go` |
| WebSocket (WS) | 长连接双向通信协议 | `internal/ws/server.go`, `internal/ws/client.go` |
| Kafka | 消息队列/流平台，用于解耦 | `internal/ws/server.go`（生产）`internal/push/consumer.go`（消费） |
| Redis | 内存 KV 存储，用于在线状态/缓存/去重 | `internal/repository/redis_repo.go` |
| MySQL | 关系数据库，持久化主数据 | `internal/repository/*.go`, `deploy/mysql/init.sql` |
| JWT | JSON Web Token，用于认证 | `internal/pkg/jwt/jwt.go`, `internal/middleware/auth.go` |
| CORS | 跨域资源共享策略 | `internal/middleware/cors.go` |
| TTL | Time To Live，键过期时间 | `SetOnline`、`SetMsgDedup`、`SetGroupMembers` |
| 幂等（Idempotency） | 重试请求不重复生效 | `msg_dedup:{msg_id}`（`redis_repo.go`） |
| ACK | 确认消息 | `internal/ws/client.go#sendAck` |

### 4.2 本项目协议术语

| 术语 | 含义 | 证据 |
|---|---|---|
| `ClientMsg` | 客户端上行 WS 包装：`type + data` | `internal/ws/protocol.go` |
| `ChatData` | 客户端聊天消息主体（`msg_id/to_id/chat_type/...`） | `internal/ws/protocol.go` |
| `ServerMsg` | 服务端下行 WS 包装 | `internal/ws/protocol.go` |
| `AckData` | ACK 数据体，核心字段 `msg_id` | `internal/ws/protocol.go` |
| `ErrorData` | WS 错误数据体（`code/msg`） | `internal/ws/protocol.go` |
| `PushMsg` | Push 服务发给 WS 内部接口的消息体 | `internal/ws/protocol.go`, `internal/push/pusher.go` |
| `KafkaChatMsg` | Kafka 中传输的聊天消息结构 | `internal/ws/protocol.go` |
| `chat_type` | 聊天类型：1 单聊，2 群聊 | `ChatData`/`KafkaChatMsg` 注释与校验 |
| `content_type` | 内容类型：1 文本，2 图片，3 文件 | `ChatData` 校验逻辑 |

### 4.3 Redis 键与状态术语

| 术语/键 | 含义 | 证据 |
|---|---|---|
| `online:{user_id}` | 用户在线状态，值是 WS 内部地址（`ws_rpc_addr`） | `internal/repository/redis_repo.go` |
| `msg_dedup:{msg_id}` | 消息幂等键，防止重复投递 | `internal/repository/redis_repo.go` |
| `group:members:{group_id}` | 群成员缓存集合（SET） | `internal/repository/redis_repo.go` |
| 条件删除在线状态 | 仅当值匹配时删除，避免旧连接删新连接状态 | `DelOnline` Lua 脚本 |
| 缓存未命中（cache miss） | Redis 未命中后回退 DB 查询 | `IsGroupMember` 返回 `cacheMiss=true` |

### 4.4 Kafka 与消费术语

| 术语 | 含义 | 证据 |
|---|---|---|
| Consumer Group | Kafka 消费组 | `config/go-im.yaml` `kafka.consumer_group` |
| Offset | 消费位点 | `internal/push/consumer.go` 手动 `CommitMessages` |
| FetchMessage | 拉取消息但不自动提交 offset | `internal/push/consumer.go` |
| 重试语义 | 处理失败不提交 offset，下次再消费 | `consumer.go` 注释与实现 |
| 坏消息跳过 | 反序列化失败提交 offset 跳过 | `consumer.go` |

### 4.5 分层术语（代码组织）

| 术语 | 在本项目中的含义 | 证据 |
|---|---|---|
| Handler 层 | HTTP 协议层：参数绑定、调用 service、返回统一响应 | `internal/handler/*.go` |
| Service 层 | 业务规则层：权限/校验/流程编排 | `internal/service/*.go` |
| Repository 层 | 数据访问层：MySQL/Redis 读写 | `internal/repository/*.go` |
| Model 层 | 持久化模型映射 | `internal/model/*.go` |
| Middleware | 横切逻辑（鉴权/CORS/日志） | `internal/middleware/*.go` |

### 4.6 容易误解的术语（重点）

| 术语 | 常见误解 | 正确认知 |
|---|---|---|
| ACK | 误以为“接收方已收到并展示” | 本项目 ACK 是 WS 网关“受理确认”，不是最终送达确认（`internal/ws/client.go`） |
| 在线用户 | 误以为只要连接过就在线 | 在线状态依赖 Redis TTL，超时不续期会过期 |
| 离线消息 | 误以为消息本体存离线表 | `offline_messages` 是索引表，消息本体在 `messages` |
| release 配置校验 | 误以为 debug/release都一样 | `Validate()` 在 release 更严格（密钥为空会拒绝启动） |

## 6. 常见误区 / 风险 / 注意点

- 不要只背术语，要把术语映射到“具体文件 + 具体函数”。
- 遇到歧义词（例如 ACK、离线消息）时，优先看实现而不是常识想当然。
- 如果后续新增术语（新消息类型、新配置项），建议同步更新本术语表。
