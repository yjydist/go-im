## 2. 核心结论

- [事实] 本项目是一个 Go 实现的 IM 后端，核心能力包括：用户/好友/群组、WebSocket 实时消息、Kafka 异步解耦、离线消息拉取。
- [事实] 运行形态是**三服务**：
  - API Server（HTTP REST，默认 `:8080`）
  - WS Gateway（WebSocket + 内部推送入口，默认 `:8081` + `:9091`）
  - Push Server（Kafka 消费 + 推送路由）
- [事实] 依赖 MySQL、Redis、Kafka，且消息主链路是 `WS -> Kafka -> Push -> WS/Offline`。
- [推断] 当前仓库定位更偏“后端教学/演示 + 可运行工程骨架”，并非完整生产 IM 功能集（例如未看到已读回执、撤回、多端同步等完整能力）。

## 4. 详细说明

### 4.1 项目解决的问题

从代码看，主要解决三类问题：

1. **关系管理**：用户注册登录、好友关系、群组成员管理（`internal/handler/*_handler.go` + `internal/service/*_service.go`）。
2. **实时消息入口**：客户端通过 WebSocket 发消息，网关做鉴权、参数校验、防重并写入 Kafka（`internal/ws/client.go`）。
3. **异步投递与离线兜底**：Push 服务消费 Kafka，先入库再路由；在线推送，离线写 `offline_messages`（`internal/push/pusher.go`）。

## 6. 常见误区 / 风险 / 注意点

- 风险 1：认为 API Server 是核心。实际上消息主通道不经过 API，而走 WS + Push。
- 风险 2：忽略 Redis 键设计（在线状态/去重/群成员缓存），会误判消息异常。
- 风险 3：把 README 里的设计图当最终真实。请始终以代码调用链为准。
