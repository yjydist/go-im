# 11-常见问题与排查

## 1. 这篇文档回答什么问题

- 项目出现问题时，应该先查哪里、再查哪里？
- 启动失败、配置错误、消息异常、构建失败分别怎么排查？
- 日志、入口、配置在仓库中的具体位置在哪里？

## 2. 核心结论

- [事实] 本项目是三服务协作（API/WS/Push），多数故障不是单点问题，排查必须按链路做：**入口 -> 配置 -> 中间件 -> 下游依赖**。
- [事实] 消息问题优先按链路分段定位：`WS 入站` -> `Kafka` -> `Push 消费` -> `WS 内部推送/离线存储`。
- [事实] 当前最常见“看起来像 bug、实际是配置”的问题是 `ws_server.internal_api_key` 与 `GOIM_WS_SERVER_INTERNAL_API_KEY` 不一致（或为空）。
- [推断] 对接手者最有效的排障方式是“先看日志关键字 + 再验证状态键/数据库落点”，而不是盲目改代码。

## 3. 关键证据

- 三服务启动入口：`cmd/api/main.go`、`cmd/ws/main.go`、`cmd/push/main.go`
- WS 消息入口与内部推送口：`internal/ws/server.go`、`internal/ws/client.go`
- Kafka 消费与重试语义：`internal/push/consumer.go`
- 推送与离线降级：`internal/push/pusher.go`
- Redis 键定义：`internal/repository/redis_repo.go`
- 离线拉取与清理：`internal/repository/message_repo.go`、`internal/service/message_service.go`
- 配置加载与覆盖：`internal/config/config.go`、`config/go-im.yaml`、`deploy/docker-config.yaml`
- 本地/容器命令：`Makefile`、`deploy/docker-compose.yaml`

## 4. 详细说明

### 4.1 故障分类总览（先看这个表）

| 问题类型 | 典型现象 | 首查位置 | 进一步排查位置 | 常见根因 |
|---|---|---|---|---|
| 启动失败 | 服务进程直接退出 | `cmd/*/main.go` 启动日志 | `internal/config/config.go` `Validate()` | 密钥为空、端口冲突、依赖不可达 |
| 环境变量问题 | 配置看似正确但行为不一致 | `internal/config/config.go` | `config/go-im.yaml` vs 实际环境 `GOIM_*` | 环境变量覆盖了 YAML |
| API 鉴权失败 | 接口返回 `10002/20005/20006` | `internal/middleware/auth.go` | `internal/pkg/jwt/jwt.go`、请求头 `Authorization` | token 过期/伪造/格式错误 |
| WS 建连失败 | `ws://.../ws?token=...` 返回 401/升级失败 | `internal/ws/server.go#HandleWS` | token 解析、Origin 检查 | token 无效、跨域 Origin 不在白名单 |
| 发消息有 ACK 但对方收不到 | 发送方看似成功，接收方无实时消息 | `internal/ws/client.go` + `internal/push/pusher.go` | Kafka 消费日志、`/internal/push` 返回码 | ACK 仅表示受理；Push 失败转离线 |
| 离线消息拉不到 | `/api/v1/message/offline` 返回空 | `internal/repository/message_repo.go#ListOffline` | `offline_messages` 与 `messages` 关联 | 未写离线表、拉取后已删除、用户ID错误 |
| 群聊发不出去 | WS 返回 `error` 403 not a group member | `internal/ws/client.go#handleChat` | Redis 群成员缓存/DB 回退 | 非群成员、缓存过期或不一致 |
| Push 不工作 | Push 服务无消费/重复重试 | `internal/push/consumer.go` | Kafka topic/group、`HandleMessage` 错误 | broker 不可达、反序列化失败、下游失败不提交 offset |
| 构建失败 | `go build` 失败 | `Makefile`、`go.mod` | Go 版本、依赖下载 | Go 版本不匹配、依赖缓存异常 |
| 测试失败 | `go test` 红 | 对应 `*_test.go` | 被测模块最近改动 | 契约变更未同步测试 |

### 4.2 分场景排查步骤

#### 场景 A：服务启动失败（最先排）

1. 看启动日志关键字：`load config failed` / `config validation failed` / `init mysql failed` / `init redis failed`。  
   证据：三入口 `cmd/*/main.go` 都有这些 `log.Fatalf`。
2. 检查配置路径是否正确：默认是 `-config config/go-im.yaml`。
3. 若 `app.env=release`，重点检查：
   - `jwt.secret` 非空且非 `changeme`
   - `ws_server.internal_api_key` 非空  
   证据：`internal/config/config.go#Validate`。
4. 检查中间件是否可达：MySQL/Redis/Kafka（本地可用 `make env-up`）。
5. Docker 场景检查 compose 服务健康状态（`deploy/docker-compose.yaml` 有 healthcheck）。

#### 场景 B：API 请求异常（401/参数错/500）

1. 先看路由是否存在（`internal/handler/router.go`）。
2. 401/Token 类错误先查 `Authorization: Bearer <token>` 格式。  
3. 参数错误（`ErrBadRequest`）看 handler 的 `ShouldBindJSON` / query 解析。  
4. 500 类错误沿 `handler -> service -> repository` 向下看，优先看 service 日志（`handleServiceError` 会记录）。
5. 检查错误码是否符合预期：`internal/pkg/errcode/errcode.go`。

#### 场景 C：WS 建连失败或连接后立刻断开

1. 检查 URL：`ws://host:8081/ws?token=...`。
2. 检查 token 是否可被 `jwt.ParseToken` 正常解析（`internal/ws/server.go`）。
3. 若配置了 `allowed_origins`，检查请求 Origin 是否在白名单（`internal/ws/server.go#NewUpgrader`）。
4. 检查 WS 服务端口是否正确（`ws_server.port`）。
5. 查看 WS 日志中的 `ws auth failed` / `ws upgrade failed`。

#### 场景 D：消息链路异常（重点）

现象：发送方收到 ACK，但接收方收不到实时消息。

按顺序排：

1. **WS 入站是否成功**：看 `message queued for kafka` 日志（`internal/ws/client.go`）。
2. **Kafka 写入是否成功**：看 `message written to kafka` 或 `kafka write failed`。
3. **Push 是否消费到**：看 `received kafka message`（`internal/push/consumer.go`）。
4. **Push 是否持久化成功**：看 `message persisted`（`internal/push/pusher.go`）。
5. **在线推送是否成功**：
   - Redis `online:{user_id}` 是否有值（`internal/repository/redis_repo.go`）
   - `/internal/push` 是否返回 200（`internal/ws/server.go#HandleInternalPush`）
6. 若在线推送失败，检查是否已降级离线：`offline_messages` 是否有记录。

> 关键提醒：当前 ACK 是网关受理确认，不是最终送达确认（见 `internal/ws/client.go` 注释与实现）。

#### 场景 E：在线推送全部失败（高频配置问题）

1. 先检查 WS 侧 `internal_api_key` 是否为空。  
   - 若为空，`/internal/push` 会直接拒绝（403）。
2. 检查 Push 侧是否使用同一个 key（`cmd/push/main.go` 把 `cfg.WSServer.InternalAPIKey` 传入 Pusher）。
3. 检查 `wsRPCAddr` 是否可达：
   - 容器内通常用 `im-ws:9091`（`deploy/docker-config.yaml`）
   - 本地默认回退 `localhost:9091`（`internal/ws/server.go`）
4. 观察 push 日志是否出现 `push request returned status ...`。

#### 场景 F：离线消息不符合预期

1. 先确认消息是否落 `messages`。
2. 再确认是否写了 `offline_messages`（`pusher.saveOffline`）。
3. 检查客户端是否已经调用过 `/api/v1/message/offline`（调用后会删除 `<=maxOfflineID`）。
4. 若怀疑竞态，重点看 `ListOffline` + `DeleteOffline` 的 `maxOfflineID` 截断逻辑。

#### 场景 G：构建/测试失败

1. 构建入口：`make build`（`go build ./cmd/...`）。
2. 测试入口：`make test`（`go test ./... -v -cover`）。
3. 常见点：
   - Go 版本低于 `go.mod` 声明（`go 1.25.0`）
   - 最近改了接口/结构体但没同步测试。

---

### 4.3 高风险区域（优先重点观察）

1. `internal/ws/client.go`：实时链路核心、分支多、并发多。
2. `internal/push/consumer.go`：offset 提交策略决定“会不会重试/丢消息”。
3. `internal/push/pusher.go`：在线/离线分流、内部推送与降级逻辑。
4. `internal/config/config.go`：配置覆盖与启动安全校验，影响全局行为。

### 4.4 建议补强项（便于后续排障）

- 给 `internal/config` 增加单元测试（覆盖 env 覆盖 + release 校验）。
- 给 `internal/ws/client.go` 增加关键分支测试（dedup、kafkaCh 满、群成员回退）。
- 给 Push 与 WS 增加统一消息追踪字段（例如贯穿 `msg_id` 的结构化日志规范）。
- 在文档中补充“本地 debug 场景如何配置 `internal_api_key`”的快速步骤。

## 6. 常见误区 / 风险 / 注意点

- 误区 1：只看 API 日志排查消息问题（消息主链路不在 API）。
- 误区 2：收到 ACK 就判断“消息已送达”。
- 误区 3：忽略环境变量覆盖，导致“我明明改了 YAML 却不生效”。
- 注意：排障时要先确认“是配置问题、依赖问题，还是代码逻辑问题”，不要一上来就改代码。
