# 03-目录结构与职责

## 1. 这篇文档回答什么问题

- 仓库里哪些目录最重要？
- 每个目录负责什么，不负责什么？
- 改需求时应该去哪一层改？

## 2. 核心结论

- [事实] 这是单模块 Go 仓库，不是 monorepo/workspace（`go.mod`，且未发现 `go.work`）。
- [事实] 目录按典型后端分层：`handler -> service -> repository -> model`，并额外有 `ws` 与 `push` 两条消息链路核心目录。
- [事实] 接手时最关键目录是：`cmd/`、`internal/ws/`、`internal/push/`、`internal/handler|service|repository/`、`config/`、`deploy/`。

## 3. 关键证据

- 入口目录：`cmd/api/main.go`, `cmd/ws/main.go`, `cmd/push/main.go`
- 业务分层：`internal/handler/*.go`, `internal/service/*.go`, `internal/repository/*.go`
- 消息链路：`internal/ws/*.go`, `internal/push/*.go`
- 配置与部署：`config/go-im.yaml`, `deploy/docker-compose.yaml`, `deploy/Dockerfile`
- 数据模型：`internal/model/*.go`, `deploy/mysql/init.sql`

## 4. 详细说明

### 4.1 顶层目录职责图

| 路径 | 负责什么 | 你什么时候会改它 |
|---|---|---|
| `cmd/` | 三个服务启动入口（初始化配置、日志、依赖、路由） | 增加服务启动参数、端口、初始化顺序、优雅停机 |
| `internal/config/` | 配置结构、加载、环境变量覆盖与安全校验 | 新增配置项、配置校验规则 |
| `internal/handler/` | HTTP 协议层（参数绑定、响应结构、路由挂载） | 改 API 入参/出参、加新路由 |
| `internal/service/` | 业务规则层 | 改业务校验、权限逻辑、流程编排 |
| `internal/repository/` | MySQL/Redis 访问层 | 改 SQL、Redis 键、缓存策略、持久化逻辑 |
| `internal/ws/` | WS 连接与上行消息处理、入 Kafka、内部推送接收 | 改实时消息协议、ACK 逻辑、去重、心跳 |
| `internal/push/` | Kafka 消费、落库、在线推送/离线存储 | 改消息路由策略、重试与降级 |
| `internal/model/` | GORM 数据结构定义 | 改字段、索引、表映射 |
| `internal/middleware/` | API 鉴权/CORS/请求日志 | 改认证规则、跨域策略 |
| `internal/pkg/` | 通用组件（jwt/errcode/response/logger/snowflake） | 统一错误码、日志策略、token 处理 |
| `config/` | 本地默认配置 | 本地调试参数、默认端口 |
| `deploy/` | 容器构建、compose、MySQL 初始化 SQL | 部署参数、环境差异、容器编排 |
| `examples/` | e2e 脚本、压测脚本、网页 demo | 联调验证、压测验证 |
| `docs/` | Swagger 与其他项目文档 | API 文档更新、学习资料补充 |

### 4.2 重点子目录说明（接手优先级）

#### A. `cmd/`（先看）

- `cmd/api/main.go`：初始化 MySQL/Redis，注册 Gin 路由，非 release 模式挂载 Swagger。
- `cmd/ws/main.go`：启动两个 HTTP server：`/ws`（对外）和 `/internal/push`（对内）。
- `cmd/push/main.go`：启动 Kafka consumer，消费消息交给 `Pusher`。

> 接手价值：你可以快速建立“服务边界 + 依赖初始化顺序”认知。

#### B. `internal/ws/`（消息入口核心）

- `server.go`：WS 鉴权升级、内部推送接口、Kafka writer 初始化。
- `client.go`：聊天上行处理、群成员校验、去重、ACK、异步写 Kafka。
- `hub.go`：在线连接管理（userID -> client）。

> 接手价值：线上“消息发不出去/重复/延迟”的大多数问题从这里入手。

#### C. `internal/push/`（消息分发核心）

- `consumer.go`：Kafka 消费循环（Fetch + 手动提交 offset）。
- `pusher.go`：先持久化再路由，在线走内部 HTTP 推送，离线写 `offline_messages`。

> 接手价值：线上“收不到消息/离线消息不对”优先查这里。

#### D. `internal/handler|service|repository/`（REST 业务面）

- `router.go` 提供 API 路由入口。
- service 层承载用户、好友、群组、历史/离线消息规则。
- repository 层封装 MySQL/Redis 访问。

> 接手价值：需求变更（字段、校验、接口行为）多发生在这三层。

### 4.3 目录边界提示（避免误改）

- `handler` 不应写 SQL；SQL/Redis 操作应落在 `repository`。
- `service` 不应处理 HTTP 细节（状态码、请求绑定）。
- `ws` 与 `push` 的消息模型在 `internal/ws/protocol.go`，改协议要联动两侧。

### 4.4 当前仓库中的“非源码产物”

- [事实] 根目录存在 `api`/`ws`/`push` 可执行文件（本地构建产物），并在 `.gitignore` 中忽略（`.gitignore` 第 45-47 行）。

## 6. 常见误区 / 风险 / 注意点

- 误区 1：只按目录名猜职责，不看调用链。
- 误区 2：在 `handler` 直接堆业务逻辑，导致不可测试。
- 风险：`internal/ws/client.go` 文件较大（核心路径集中），改动要配套回归测试。
