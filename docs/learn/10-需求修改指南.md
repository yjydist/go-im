# 10-需求修改指南

## 1. 这篇文档回答什么问题

- 如果我要开始改需求，应该先看哪里？
- 不同类型需求通常改哪些文件？
- 改完后要联动检查哪些位置，避免“改一处坏一片”？

## 2. 核心结论

- [事实] 本仓库改动入口主要分三类：
  1) REST 业务链（`handler -> service -> repository`）；
  2) 实时消息链（`internal/ws` + `internal/push`）；
  3) 配置与部署链（`internal/config` + `config/` + `deploy/`）。
- [事实] 消息相关改动最容易扩散：`internal/ws/protocol.go`、`internal/ws/client.go`、`internal/push/pusher.go`、`internal/push/consumer.go` 往往要联动。
- [事实] 本仓库没有业务前端工程（未发现 `web/`、`frontend/`、`*.tsx` 等），所以“改页面展示”通常不在本仓库主链完成；仅有 `examples/chat.html` 用于调试演示。
- [推断] 对接手者最稳妥的策略是：**先确认入口，再画调用链，再做最小改动，再按联动清单回归**。

## 3. 关键证据

- 路由与入口：`internal/handler/router.go`、`cmd/api/main.go`、`cmd/ws/main.go`、`cmd/push/main.go`
- REST 分层：`internal/handler/*.go`、`internal/service/*.go`、`internal/repository/*.go`
- 实时链路：`internal/ws/server.go`、`internal/ws/client.go`、`internal/ws/protocol.go`
- 推送链路：`internal/push/consumer.go`、`internal/push/pusher.go`
- 数据模型：`internal/model/*.go`、`deploy/mysql/init.sql`
- 配置体系：`internal/config/config.go`、`config/go-im.yaml`、`deploy/docker-config.yaml`、`deploy/docker-compose.yaml`
- 测试入口：`Makefile`（`make test`）、`internal/**/*_test.go`

## 4. 详细说明

### 4.1 常见需求类型 -> 修改入口 -> 联动检查

| 需求类型 | 首先查看的位置 | 常见修改文件 | 联动检查点 | 风险说明 |
|---|---|---|---|---|
| 新增 API 字段（入参/出参） | `internal/handler/*_handler.go` | Handler 请求结构体、Service 入参、Repository 查询/写入、`docs/docs.go`（Swagger） | `router.go` 路由；`internal/pkg/response` 统一响应；对应 handler/service/repo 测试 | 字段只改一层会出现“编译过了但语义错” |
| 调整接口业务规则（权限、校验、状态限制） | `internal/service/*_service.go` | Service 规则、必要时 handler 参数校验、repository 条件 | 业务错误码（`internal/pkg/errcode/errcode.go`）；`service/errors.go` 解析逻辑；测试断言 | 容易把业务错误返回成内部错误 |
| 调整 SQL 查询/落库行为 | `internal/repository/*.go` | `GetHistory`/`ListOffline`/`ListMembers` 等 Repository 方法 | `internal/model/*.go`、`deploy/mysql/init.sql`、相关 service 测试 | SQL 条件变更会影响分页、离线删除边界 |
| 新增/调整数据库字段 | `internal/model/*.go` + `deploy/mysql/init.sql` | 模型结构体 tag、SQL 初始化脚本、Repository 读写 | 若使用 Docker 初始化，需重建库验证；API 返回结构和 Swagger 同步 | [事实] 运行时代码未做 AutoMigrate（仅测试中有），漏改 SQL 会导致运行不一致 |
| 调整 WebSocket 消息协议 | `internal/ws/protocol.go` | `ClientMsg/ChatData/KafkaChatMsg/ServerMsg` | `internal/ws/client.go` 解析逻辑；`internal/push/consumer.go` 反序列化；`examples/e2e/main.go`、`examples/bench/main.go` | 高风险：协议两端不同步会直接消息不可用 |
| 调整消息去重/ACK 语义 | `internal/ws/client.go` | `handleChat`、`SetMsgDedup`、`sendAck`、`kafkaWorker` | Push 侧重试行为（`consumer.go`）；客户端对 ACK 语义预期；e2e 脚本 | ACK 目前是“网关受理确认”，不是“最终送达确认” |
| 调整群聊成员校验/缓存策略 | `internal/ws/client.go`、`internal/push/pusher.go`、`internal/repository/redis_repo.go` | `IsGroupMember`、`checkGroupMemberFromDB`、`SetGroupMembers` | `group_service.JoinGroup` 的缓存清理；`push.group_member_cache_ttl_sec` 配置 | WS 与 Push 缓存 TTL 来源不同，容易出现行为不一致 |
| 调整在线推送/离线策略 | `internal/push/pusher.go` | `pushToUser`、`sendPush`、`saveOffline` | WS 内部接口鉴权（`internal/ws/server.go`）；Redis 在线状态键 | 若 `internal_api_key` 不一致，在线推送会降级为离线 |
| 调整配置项/环境变量行为 | `internal/config/config.go` | Config 结构体、`Validate()`、`config/*.yaml`、`deploy/*.yaml` | 三个 `cmd/*/main.go` 启动校验；Compose 环境映射；文档同步 | 配置改动是跨服务影响，建议先最小化验证 |
| 改页面展示 | 当前主业务不适用 | 若仅调试页，改 `examples/chat.html` | 需确认该页面是否仅 demo，不是正式客户端 | [事实] 仓库未发现前端应用代码；此类需求通常在外部客户端仓库完成 |

### 4.2 高风险改动清单（优先谨慎）

1. **改 `internal/ws/client.go` 的 `handleChat` 与 `kafkaWorker`**  
   原因：同时影响校验、去重、ACK、Kafka 入队与回滚。
2. **改消息协议结构（`protocol.go`）**  
   原因：WS 网关、Push 消费、示例脚本都依赖同一协议。
3. **改 `/internal/push` 鉴权和路由地址配置**  
   原因：`internal_api_key` 或 `rpc_advertise_addr` 配错会导致在线推送全失败。
4. **改离线消息删除逻辑（`ListOffline`/`DeleteOffline`）**  
   原因：并发下容易误删新到达离线消息。
5. **改 `service.ParseBusinessError` 规则**  
   原因：错误码解析机制影响全局响应码行为。

### 4.3 修改后联动检查清单（建议逐项打勾）

#### A. 编译与基础回归

- [ ] `go test ./... -cover` 通过（或至少受影响包测试通过）
- [ ] 三服务都能启动（`make run-api` / `make run-ws` / `make run-push`）
- [ ] 关键日志无持续 error/warn（关注 WS、Push）

#### B. 契约一致性

- [ ] 路由、请求参数、响应结构与文档一致（`router.go` + Swagger）
- [ ] WebSocket 协议结构体改动已联动 Push 反序列化
- [ ] 错误码语义未破坏（`errcode` + handler/service 测试）

#### C. 数据一致性

- [ ] MySQL 模型与 SQL 初始化脚本一致
- [ ] Redis 键名/TTL 改动已联动 WS 与 Push 两侧
- [ ] 离线消息拉取后删除逻辑仍符合竞态安全设计

#### D. 配置与部署一致性

- [ ] `config/go-im.yaml` 与 `deploy/docker-config.yaml` 同步
- [ ] `GOIM_*` 覆盖行为验证过（至少验证关键密钥/DSN）
- [ ] 如改动影响启动参数/流程，已同步文档

### 4.4 建议的“改需求起手式”（给第一次接手者）

1. 先定位需求属于哪一类（参考 4.1 表格）。
2. 从**入口文件**开始读（路由/WS 入口/consumer 入口），画 1 条调用链。
3. 只改最短路径文件，不顺手重构无关模块。
4. 先补/改测试，再改实现（至少覆盖关键分支）。
5. 跑最小回归：登录 -> 发消息 -> 拉离线（或对应业务链）。

## 6. 常见误区 / 风险 / 注意点

- 误区 1：只改 handler 不改 service/repository，导致“接口成功但业务不生效”。
- 误区 2：只改 WS 不改 Push（或反过来），导致协议不兼容。
- 误区 3：改了 YAML 忘了环境变量覆盖，线上行为与本地不一致。
- 注意：消息链路问题往往是跨模块问题，排查要同时看 WS + Push + Redis + Kafka。
