# 09-测试与质量保障

## 1. 这篇文档回答什么问题

- 项目目前有哪些测试？覆盖到哪里？
- 质量保障薄弱点在哪？

## 2. 核心结论

- [事实] 项目以 Go 单元测试为主，测试文件主要集中在 `internal/*`。
- [事实] 已执行 `go test ./... -cover`，核心包覆盖率高低差异明显：
  - `internal/handler` 90.6%
  - `internal/middleware` 92.4%
  - `internal/service` 77.5%
  - `internal/repository` 72.9%
  - `internal/push` 45.1%
  - `internal/ws` 13.1%
  - `internal/config` 0%
- [事实] 当前未发现 CI 配置文件，质量检查主要依赖本地执行。

## 3. 关键证据

- 测试命令：`Makefile` (`go test ./... -v -cover`)
- 测试文件分布：`internal/**/*_test.go`
- 关键测试样例：
  - handler：`internal/handler/handler_test.go`
  - service：`internal/service/service_test.go`
  - repository：`internal/repository/*_test.go`
  - push：`internal/push/pusher_test.go`
  - ws：`internal/ws/hub_test.go`, `internal/ws/protocol_test.go`

## 4. 详细说明

### 4.1 测试类型与工具

| 类型 | 实现方式 | 说明 |
|---|---|---|
| 单元测试 | `go test` | 标准 testing 框架 |
| 轻集成测试 | SQLite 内存库 + miniredis | service/repository/handler 测试中使用 |
| 协议测试 | JSON round-trip / WS hub 行为 | `internal/ws/*_test.go` |

### 4.2 覆盖情况解读

- **强覆盖区**：`handler`、`middleware`、`pkg/*`
  - 参数校验、错误码返回、Auth/CORS 中间件行为覆盖较好。
- **中覆盖区**：`service`、`repository`
  - 主要业务规则与数据访问路径有测试，但仍偏 happy-path。
- **薄弱区**：`ws/client.go`、`push/consumer.go`、`config`
  - WS 核心上行逻辑与 Kafka 写入回滚、consumer 重试行为缺少深入测试。

### 4.3 当前质量保障缺口

1. 缺少 CI 自动执行测试与质量门禁。
2. 缺少针对 `internal/config` 的单测（配置覆盖、校验边界）。
3. `internal/ws/client.go` 是关键路径但覆盖率偏低。
4. 缺少端到端自动化回归（`examples/e2e` 是脚本，不是 pipeline 中的自动测试）。

### 4.4 建议最小补强优先级

1. 给 `internal/config` 增加单元测试（环境变量覆盖、release 校验）。
2. 为 `internal/ws/client.go` 的核心分支补测试：
   - dedup hit/miss
   - kafkaCh 满时回滚
   - 群成员缓存 miss -> DB 回退
3. 为 `internal/push/consumer.go` 增加异常分支测试（反序列化失败、处理失败不提交 offset）。

## 5. 如何继续深入

- 结合 [10-需求修改指南](./10-需求修改指南.md) 在每类需求里补“对应测试点”。
- 结合 [11-常见问题与排查](./11-常见问题与排查.md) 将高频故障转化为回归测试。

## 6. 常见误区 / 风险 / 注意点

- 误区：总覆盖率看起来不低就等于关键路径安全。
- 风险：WS/Push 高并发行为主要靠代码逻辑保障，测试保护还不够。
- 注意：覆盖率数字会随本地环境和测试参数变化，建议以 CI 固化（当前仓库未内建）。
