# 05-核心运行流程

## 1. 这篇文档回答什么问题

- 请求/消息从哪里进入系统？
- 中间经过哪些关键节点？
- 数据写到哪里？异常怎么处理？

## 2. 核心结论

- [事实] 最关键链路是：**WS 上行消息 → Kafka → Push 消费 → MySQL 持久化 → 在线推送或离线存储**。
- [事实] REST 侧主要负责账号与关系数据管理，以及离线/历史消息拉取。
- [事实] 离线消息拉取后采用 `maxOfflineID` 截断删除，避免并发误删新消息。

## 3. 关键证据

- 路由入口：`internal/handler/router.go`
- WS 入口：`internal/ws/server.go` `HandleWS`
- 上行处理：`internal/ws/client.go` `handleChat`
- Kafka 消费：`internal/push/consumer.go` `Start`
- 推送路由：`internal/push/pusher.go` `HandleMessage`
- 离线拉取与删除：`internal/service/message_service.go`, `internal/repository/message_repo.go`

## 4. 详细说明

---

### 流程 A：用户登录（HTTP）

**触发点**：`POST /api/v1/user/login`  
**入口文件**：`internal/handler/user_handler.go#Login`

#### 调用链

1. `router.RegisterRoutes` 注册 `/api/v1/user/login`。
2. `UserHandler.Login` 做参数绑定。
3. `UserService.Login`：
   - 查用户（`UserRepository.GetByUsername`）
   - 校验密码（bcrypt）
   - 生成 JWT（`internal/pkg/jwt.GenerateToken`）
4. 返回 `{code,msg,data.token}`。

#### 状态变化

- 不写 Redis 在线状态（在线状态由 WS 建连时写入）。

#### 失败分支

- 用户不存在 / 密码错误 / 用户封禁：返回业务错误码（`service.ErrBusiness` + `errcode`）。

---

### 流程 B：WebSocket 单聊发送主链路

**触发点**：客户端向 `ws://host:8081/ws?token=...` 建连后发送 `{"type":"chat"}`。  
**入口文件**：`internal/ws/server.go#HandleWS`，`internal/ws/client.go#handleChat`

#### 调用链（从入口到落点）

1. `HandleWS` 校验 token，升级连接，创建 `Client`。
2. `Client.Start`：启动 `readPump/writePump/kafkaWorker`，并 `SetOnline(user_id, wsRPCAddr)`。
3. `readPump` 收到消息，进入 `handleMessage -> handleChat`。
4. `handleChat` 做输入校验（msg_id/chat_type/to_id/content）。
5. Redis 幂等：`SetMsgDedup(msg_id, 5m)`。
6. 将 `KafkaChatMsg` 放入 `kafkaCh`，**立即 ACK**。
7. `kafkaWorker` 异步写 Kafka。
8. Push 服务 `Consumer.Start` Fetch 消息并反序列化。
9. `Pusher.HandleMessage` 先写 MySQL `messages`。
10. `pushToUser`：
    - Redis 查 `online:{to_id}`
    - 在线：调用 `http://{ws_addr}/internal/push`
    - 不在线或推送失败：写 `offline_messages`

#### 输入/输出

- 输入：`ChatData{msg_id,to_id,chat_type,content_type,content}`
- 输出（发送方）：`ack` 或 `error`
- 输出（接收方）：在线收到 `chat`；离线走 API 拉取

#### 异常处理

- Kafka 写失败：回滚 dedup key（允许客户端重试）
- `/internal/push` 失败：降级为离线存储
- consumer 处理失败：不提交 offset，等待下一次重试

#### 日志点

- WS：`message queued for kafka`, `message written to kafka`（`internal/ws/client.go`）
- Push：`received kafka message`, `message persisted`（`internal/push/*`）

---

### 流程 C：群聊发送链路（含成员校验）

**触发点**：`chat_type=2` 发送到群。  
**入口文件**：`internal/ws/client.go#handleChat`

#### 关键节点

1. 先查 Redis `IsGroupMember(group_id, user_id)`。
2. 缓存未命中时回退 DB：`GroupRepository.GetMember`。
3. 若 DB 命中，异步回填整个群成员缓存。
4. Push 侧消费后 `pushToGroup`：
   - 先读 Redis 群成员
   - miss 时查 DB `ListMemberIDs` 并回填
   - 并发扇出推送（可配置并发度）

#### 风险点

- WS 回填 TTL 固定 30 分钟；Push 回填 TTL 可配置，存在策略不一致风险。

---

### 流程 D：离线消息拉取与清理

**触发点**：`GET /api/v1/message/offline`。  
**入口文件**：`internal/handler/message_handler.go#GetOfflineMessages`

#### 调用链

1. Auth 中间件获取 user_id。
2. `MessageService.GetOfflineMessages` 调 `MessageRepository.ListOffline`。
3. 读取离线消息时先取 `maxOfflineID`。
4. 返回消息后，执行 `DeleteOffline(user_id, id<=maxOfflineID)`。

#### 为什么这样设计

- [事实] 这样可避免“拉取期间新到达消息被误删”的竞态。

## 6. 常见误区 / 风险 / 注意点

- 误区：ACK=送达。当前 ACK 是“网关受理确认”。
- 风险：同时改 `internal/ws/protocol.go` 与 `examples/*` 但忘记改 Push 反序列化会引入兼容问题。
- 注意：消息链路问题排查要同时看 WS 与 Push 日志，不能只看 API。
