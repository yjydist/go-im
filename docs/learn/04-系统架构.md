# 04-系统架构

## 1. 这篇文档回答什么问题

- 系统由哪些层组成？
- 模块边界和依赖关系是什么？
- 哪些点最复杂、最容易改坏？

## 2. 核心结论

- [事实] 系统可分为三条主线：
  1) REST 业务线（`handler -> service -> repository`）；
  2) WS 实时入口线（连接管理 + 校验 + Kafka 入队）；
  3) Push 异步投递线（Kafka 消费 + 落库 + 在线/离线分流）。
- [事实] Redis 承担三类状态：在线状态、消息去重、群成员缓存（`internal/repository/redis_repo.go`）。
- [事实] MySQL 是消息与关系数据最终落点，Kafka 是异步削峰解耦层。
- [推断] 当前架构强调“可运行和清晰链路”，但在配置一致性、测试覆盖深度、权限细粒度方面仍有加强空间。

## 3. 关键证据

- 架构入口：`cmd/api/main.go`, `cmd/ws/main.go`, `cmd/push/main.go`
- REST 分层：`internal/handler/router.go`, `internal/service/*.go`, `internal/repository/*.go`
- WS 入口与内部推送：`internal/ws/server.go`, `internal/ws/client.go`, `internal/ws/hub.go`
- Push 消费与路由：`internal/push/consumer.go`, `internal/push/pusher.go`
- Redis 状态键：`internal/repository/redis_repo.go`

## 4. 详细说明

### 4.1 架构分层（按职责，而非按目录名硬套）

| 层 | 模块 | 主要职责 |
|---|---|---|
| 接入层 | `cmd/*` | 启动服务、初始化依赖、装配运行时 |
| 协议层 | `internal/handler`, `internal/ws/server.go` | 接收 HTTP/WS 请求、做协议处理 |
| 业务层 | `internal/service`, `internal/ws/client.go`, `internal/push/pusher.go` | 承载业务规则、状态判断、路由决策 |
| 数据访问层 | `internal/repository` | 统一访问 MySQL/Redis |
| 数据模型层 | `internal/model` | 表结构映射 |
| 基础能力层 | `internal/pkg/*`, `internal/middleware/*` | JWT、日志、错误码、响应封装、中间件 |

### 4.2 模块关系（核心依赖方向）

1. **API Server**  
   `router -> handler -> service -> repository -> MySQL/Redis`

2. **WS Gateway**  
   `HandleWS -> Client.handleChat -> Redis(dedup/member check) -> Kafka writer`

3. **Push Server**  
   `Kafka consumer -> Pusher.HandleMessage -> MessageRepo.Create -> online check -> /internal/push or offline_messages`

### 4.3 三个关键架构设计点

#### 设计点 A：消息入口与投递解耦

- [事实] WS 不直接做重逻辑推送，只负责校验/入队/ACK；Push 负责消费后落库与分发。  
  证据：`internal/ws/client.go`（写 Kafka、发送 ACK）、`internal/push/pusher.go`（落库+推送）。

#### 设计点 B：多层幂等与重试语义

- [事实] WS 侧 `SETNX msg_dedup:{msg_id}` 防重；Kafka 写失败会回滚 dedup key，允许客户端重试。  
  证据：`internal/ws/client.go`（`SetMsgDedup` / `DelMsgDedup`）。

#### 设计点 C：在线状态与路由地址绑定

- [事实] Redis `online:{user_id} -> ws_rpc_addr` 保存“用户在哪个 WS 节点”，Push 按地址调用该节点 `/internal/push`。  
  证据：`internal/repository/redis_repo.go`、`internal/push/pusher.go`、`internal/ws/server.go`。

### 4.4 架构风险与复杂点

1. **WS 客户端逻辑集中**（`internal/ws/client.go`）  
   - 风险：文件大、职责多（校验/鉴权后的业务校验/缓存回填/Kafka 异步/ACK），改动容易牵一发而动全身。

2. **ACK 语义易误解**  
   - [事实] ACK 在消息进入 Kafka 异步队列阶段就返回，不等同于最终送达。  
   - 影响：前端若把 ACK 当“送达”会造成状态错误。

3. **缓存 TTL 策略存在双来源**  
   - [事实] Push 侧群成员缓存 TTL 可配置（`push.group_member_cache_ttl_sec`），WS 回填写死 `30*time.Minute`。  
   - 风险：行为不一致，排查缓存问题时易混淆。

4. **权限边界较粗**  
   - [事实] `GET /api/v1/group/members` 当前仅鉴权，不校验调用者是否群成员。  
   - 证据：`internal/handler/group_handler.go` + `internal/service/group_service.go`。

## 6. 常见误区 / 风险 / 注意点

- 误区：把架构图当真实行为。应以函数调用链与配置覆盖为准。
- 注意：凡是改动 `protocol.go`、`client.go`、`pusher.go`，都属于高风险改动。
