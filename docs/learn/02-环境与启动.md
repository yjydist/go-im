# 02-环境与启动

## 1. 这篇文档回答什么问题

- 本地怎么把项目跑起来？
- 有哪些必备依赖、端口和环境变量？
- 启动失败时优先查哪里？

## 2. 核心结论

- [事实] 项目支持两种启动方式：
  1) `make docker-up` 一键拉起中间件+三服务；
  2) `make env-up` 后分别启动 `run-api/run-ws/run-push`。
- [事实] 三服务默认端口：API `8080`、WS `8081`、WS 内部推送口 `9091`。
- [事实] 配置文件默认路径是 `config/go-im.yaml`，可通过 `-config` 覆盖（见三入口 `main.go`）。
- [事实] 配置支持 `GOIM_*` 环境变量覆盖（Viper 自动映射）。

## 3. 关键证据

- 启动命令：`Makefile`
- 默认配置路径：`cmd/api/main.go`、`cmd/ws/main.go`、`cmd/push/main.go`（`-config` flag）
- 环境变量覆盖：`internal/config/config.go`
- 开发配置：`config/go-im.yaml`
- Docker 配置与环境注入：`deploy/docker-compose.yaml`、`deploy/docker-config.yaml`、`deploy/.env.example`

## 4. 详细说明

### 4.1 前置条件

| 条件 | 说明 | 证据 |
|---|---|---|
| Go 环境 | `go.mod` 声明 `go 1.25.0` | `go.mod` |
| Docker / Compose | 用于中间件和一键部署 | `Makefile`, `deploy/docker-compose.yaml` |
| 中间件 | MySQL / Redis / Kafka | `deploy/docker-compose.yaml` |

### 4.2 启动方式 A：Docker 一键（推荐快速体验）

```bash
make docker-up
```

停止：

```bash
make docker-down
```

说明：会按 `deploy/docker-compose.yaml` 构建并启动 `im-api / im-ws / im-push` 以及 MySQL/Redis/Kafka。

### 4.3 启动方式 B：本地开发分服务启动

1) 启动中间件：

```bash
make env-up
```

2) 新开三个终端分别运行：

```bash
make run-api
make run-ws
make run-push
```

3) 收尾：

```bash
make env-down
```

### 4.4 关键端口

| 端口 | 作用 | 证据 |
|---|---|---|
| 8080 | API HTTP | `config/go-im.yaml` `api_server.port` |
| 8081 | WebSocket 外部入口 | `config/go-im.yaml` `ws_server.port` |
| 9091 | WS 内部推送入口（`/internal/push`） | `config/go-im.yaml` `ws_server.rpc_port`, `cmd/ws/main.go` |
| 3306 | MySQL | `deploy/docker-compose.yaml` |
| 6379 | Redis | `deploy/docker-compose.yaml` |
| 9092 | Kafka 对宿主机端口 | `deploy/docker-compose.yaml` |

### 4.5 关键环境变量

| 变量名 | 用途 | 在哪里被使用 |
|---|---|---|
| `GOIM_JWT_SECRET` | 覆盖 `jwt.secret` | `internal/config/config.go` + compose 注入 |
| `GOIM_MYSQL_DSN` | 覆盖 MySQL DSN | 同上 |
| `GOIM_WS_SERVER_INTERNAL_API_KEY` | 保护 `/internal/push` | `internal/ws/server.go` 校验 |

> 说明：`deploy/.env.example` 给的是 Compose 外层变量（`JWT_SECRET`、`MYSQL_ROOT_PASSWORD`、`INTERNAL_API_KEY`），再由 compose 映射到 `GOIM_*`。

> 重要提示（容易踩坑）：
> - [事实] `internal/ws/server.go` 中如果 `internal_api_key` 为空，会拒绝所有 `/internal/push` 请求（403）。
> - [事实] 默认 `config/go-im.yaml` 里该值为空，若你本地分服务启动并希望“在线实时推送”生效，需要显式设置 `GOIM_WS_SERVER_INTERNAL_API_KEY`（并让 Push 读到同一个值）。

### 4.6 首次启动建议验证

1. API：访问 `http://localhost:8080/swagger/index.html`（非 release 模式下可用）。
2. WS：使用 `examples/chat.html` 或自定义客户端连接 `ws://localhost:8081/ws?token=...`。
3. Push：观察 push 服务日志是否出现 `kafka consumer started`（`internal/push/consumer.go`）。

## 6. 常见误区 / 风险 / 注意点

- 风险 1：只启动 API，不启动 WS/Push，就会出现“登录正常但消息链路不通”。
- 风险 2：`release` 模式下密钥为空会被 `cfg.Validate()` 拒绝启动（`internal/config/config.go`）。
- 风险 3：`/internal/push` 依赖 `internal_api_key`，不匹配时会返回 401（`internal/ws/server.go`）。
