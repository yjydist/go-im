# 06-核心模块详解

## 1. 这篇文档回答什么问题

- 哪些模块是“必须懂”的核心模块？
- 每个模块的入口、依赖、输出、风险点在哪里？

## 2. 核心结论

- [事实] 对接手价值最高的模块是：
  1) `internal/ws`（实时入口）；
  2) `internal/push`（异步路由）；
  3) `handler/service/repository`（业务 API 面）；
  4) `internal/config`（运行行为开关）。
- [事实] 最容易改坏的是 `internal/ws/client.go` 与 `internal/push/pusher.go`。

## 3. 关键证据

- `internal/ws/client.go`, `internal/ws/server.go`, `internal/ws/hub.go`
- `internal/push/consumer.go`, `internal/push/pusher.go`
- `internal/handler/router.go`, `internal/service/*.go`, `internal/repository/*.go`
- `internal/config/config.go`

## 4. 详细说明（按模块模板）

---

## 模块 A：`internal/ws`（WebSocket 网关）

- **模块职责**：处理 WS 建连鉴权、在线状态、上行消息校验、幂等、Kafka 入队、下行 ACK/错误。
- **为什么重要**：消息入口都在这里，任何协议变更、实时链路问题都绕不过它。
- **入口位置**：`cmd/ws/main.go` -> `ws.NewServer` -> `HandleWS`。
- **关键文件**：
  - `server.go`：建连与内部推送入口
  - `client.go`：`handleChat`、`kafkaWorker`
  - `hub.go`：在线连接管理
  - `protocol.go`：协议结构
- **对外暴露**：
  - `/ws?token=...`
  - `/internal/push`（内部 HTTP）
- **依赖哪些模块**：`repository.RedisRepository`、`repository.GroupRepository`、Kafka writer、`pkg/jwt`。
- **被哪些模块依赖**：Push 服务通过 `/internal/push` 调用。
- **依赖外部系统**：Redis、Kafka（以及群成员 DB 回查时依赖 MySQL）。
- **核心数据结构**：`ClientMsg`, `ChatData`, `KafkaChatMsg`, `ServerMsg`。
- **核心执行流程**：见 [05-核心运行流程](./05-核心运行流程.md) 流程 B/C。
- **配置项**：`ws_server.*`（端口、RPC 地址、内部 key、心跳与超时参数、allowed_origins）。
- **错误处理方式**：协议错误返回 WS `error` 消息；Kafka 写失败回滚 dedup key。
- **测试覆盖情况**：
  - 已覆盖：`hub_test.go`、`protocol_test.go`
  - 薄弱：`client.go` 主逻辑、`server.go` HTTP 入口缺少系统级测试
- **常见修改点**：新增消息类型、调整 ACK 语义、调整去重 TTL、调整群聊校验。
- **风险点**：
  - `client.go` 逻辑密度高；
  - ACK 与最终送达语义不一致；
  - 缓存策略分散（WS 与 Push）。
- **相关文档**：
  - [04-系统架构](./04-系统架构.md)
  - [05-核心运行流程](./05-核心运行流程.md)
  - [11-常见问题与排查](./11-常见问题与排查.md)

## 模块 B：`internal/push`（Kafka 消费与推送）

- **模块职责**：消费 Kafka 消息、持久化消息、在线推送、离线存储。
- **为什么重要**：决定“消息最终如何到达用户”。
- **入口位置**：`cmd/push/main.go` -> `consumer.Start(ctx)`。
- **关键文件**：`consumer.go`, `pusher.go`。
- **对外暴露**：无外部 HTTP；对 Kafka topic `chat_messages` 进行消费。
- **依赖哪些模块**：Message/Group/Redis Repository；WS 内部推送接口。
- **被哪些模块依赖**：WS 写 Kafka 后由它继续处理。
- **依赖外部系统**：Kafka、MySQL、Redis、WS 内部 HTTP。
- **核心数据结构**：`ws.KafkaChatMsg`, `model.Message`, `model.OfflineMessage`。
- **核心执行流程**：FetchMessage -> 反序列化 -> HandleMessage -> Commit offset。
- **配置项**：`kafka.*`, `push.*`, `ws_server.internal_api_key`。
- **错误处理方式**：
  - 反序列化失败：提交 offset 跳过坏消息；
  - HandleMessage 失败：不提交 offset，等待重试；
  - 在线推送失败：降级离线存储。
- **测试覆盖情况**：`pusher_test.go` 覆盖单聊在线/离线、群聊扇出、推送失败降级；`consumer.go` 目前缺少直接单测。
- **常见修改点**：推送策略、扇出并发、离线条件、Kafka 参数。
- **风险点**：
  - offset 提交窗口可能导致重复消费；
  - ws_addr 不可达会触发离线降级。
- **相关文档**：
  - [05-核心运行流程](./05-核心运行流程.md)
  - [10-需求修改指南](./10-需求修改指南.md)

## 模块 C：`handler + service + repository`（REST 业务链）

- **模块职责**：提供用户/好友/群组/消息查询接口。
- **为什么重要**：所有“非实时”功能都从这里改。
- **入口位置**：`internal/handler/router.go`
- **关键文件**：
  - Handler：`*_handler.go`
  - Service：`*_service.go`
  - Repository：`*_repo.go`
- **对外暴露**：`/api/v1/*` 路由。
- **依赖哪些模块**：`middleware.Auth`, `pkg/response`, `pkg/errcode`。
- **被哪些模块依赖**：客户端 REST 调用直接依赖。
- **核心数据结构**：`model.User/Friendship/Group/Message/OfflineMessage`。
- **配置项**：`jwt.secret`, `api_server.allowed_origins`, `mysql/redis`。
- **错误处理方式**：service 抛业务错误码，handler 统一翻译为响应。
- **测试覆盖情况**：
  - `handler_test.go` 覆盖较完整；
  - `service_test.go`、`repository_test.go` 覆盖主要业务。
- **常见修改点**：参数校验、业务规则、SQL 查询条件、返回字段。
- **风险点**：
  - `ParseBusinessError` 通过字符串末尾提取错误码，维护性一般；
  - 部分接口权限校验粒度较粗（如群成员列表查询）。

---

## 模块 D：`internal/config`（配置中心）

- **模块职责**：读取 YAML、映射到结构体、应用 `GOIM_*` 覆盖、执行安全校验。
- **入口位置**：三个 `cmd/*/main.go` 在启动最早阶段调用 `config.Load` 与 `cfg.Validate`。
- **关键文件**：`internal/config/config.go`, `config/go-im.yaml`, `deploy/docker-config.yaml`。
- **对外暴露**：`config.GlobalConfig`（全局读取）。
- **风险点**：
  - 当前 `internal/config` 没有测试覆盖（`go test ./... -cover` 结果 0%）；
  - `refresh_expire_hours` 在仓库中未看到实际使用代码。

## 6. 常见误区 / 风险 / 注意点

- 不要直接跨层调用（例如 handler 直接用 repository）。
- 改协议时一定联动 `ws/protocol.go`、push 反序列化、examples。
- 改 Redis 键策略时要联动 WS 和 Push 两侧逻辑。
