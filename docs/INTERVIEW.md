# go-im 面试问答与追问链（INTERVIEW）

## 0) 1 分钟项目介绍（可背框架，不要死记）

这是一个 Go 实现的 IM 后端，拆成 API、WS、Push 三个服务。客户端通过 WebSocket 连接 WS 网关发消息，网关做鉴权、参数校验和幂等后写入 Kafka；Push 服务异步消费后先持久化消息，再按 Redis 在线状态决定实时推送还是离线存储。API 服务提供用户、好友、群组、离线拉取和历史查询，整体目标是把连接层和重逻辑解耦，兼顾实时性与系统稳定性。

---

## 1) 高频问题 + 参考回答

### Q1：为什么要拆成三个服务？

- 参考答：
  - WS 连接层关注高并发长连接与快速 ACK，不做重逻辑。
  - Push 层处理持久化与路由，避免网关阻塞。
  - API 层承载 CRUD 与查询能力，职责清晰，部署可独立扩缩。

- 追问链：
  1. 如果不拆会出现什么问题？
  2. 拆分后的一致性与故障处理成本如何增加？

### Q2：你们 ACK 的语义是什么？

- 参考答：
  - 当前 ACK 是“网关已受理并排入 Kafka 发送队列”的确认，不等同于“对端已收到”。
  - Kafka 写失败时会回滚 dedup key，允许客户端重试。

- 追问链：
  1. 如何实现“送达 ACK / 已读 ACK”？
  2. 当前语义下客户端要如何做重试策略？

### Q3：如何防重复消息？

- 参考答：
  - 客户端提供 `msg_id`，WS 用 Redis `SETNX msg_dedup:{msg_id}` 实现幂等窗口。
  - 如果 Kafka 写入失败，删除 dedup key，避免消息永久卡死不可重发。

### Q4：离线消息为什么按 `maxOfflineID` 删除？

- 参考答：
  - 先读取当前最大离线记录 ID，再拉取 `<= maxOfflineID` 的消息并删除同范围。
  - 避免拉取期间新到离线消息被误删（并发竞态规避）。

### Q5：群聊如何做权限与性能平衡？

- 参考答：
  - 发送前先校验群成员资格。
  - 优先 Redis 群成员缓存，未命中回退 DB，并异步回填缓存。
  - 推送阶段并发扇出并限制并发度，防止 goroutine 失控。

## 2.5) “代码不是我写的”如何诚实回答（适配当前情况）

可用回答框架：

1. **先承认边界**：这个项目最初由 AI/他人生成，我不是最初作者。
2. **再说明你做了什么**：我重点做了主链路梳理、模块理解、问题定位与维护验证。
3. **给出代码证据**：能解释 `internal/ws/client.go` 的幂等与 ACK 语义、`internal/push/pusher.go` 的在线/离线分流。
4. **强调可维护能力**：我现在可以独立定位消息链路问题并提出改进方案。

不建议说法：

- “这个系统完全是我从零设计实现的”。
